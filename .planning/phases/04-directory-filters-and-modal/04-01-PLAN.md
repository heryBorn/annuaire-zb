<objective>
Refactor DirectoryPage so member data loads once (via useMemberFetch hook) and filtered results are derived synchronously via useMemo. This eliminates the search-clear race condition (the flushSync bug), makes all four dropdowns immediately reactive after first load, and wires the stats counters to real data.

Purpose: The current runSearch() pattern fetches fresh on every click and uses flushSync to force DOM clearing — this is the wrong tool for the job (flushSync is for synchronous DOM measurement, not async coordination). The correct fix is architectural: separate data loading (async, happens once) from filter application (synchronous, derived).
Output: A refactored DirectoryPage.jsx that imports useMemberFetch, derives filteredResults via useMemo, and passes an onClick stub to MemberCard (to be wired in Plan 03).
</objective>
